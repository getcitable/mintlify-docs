---
title: "Hard rules"
description: "Platform invariants every executor obeys. Violating any of these is a bug, not a preference."
---

These are enforced at the database, service, or tool layer. Code that violates any of them is the bug; the rule is the contract.

## 1. `intent` is `engagement` or `commercial` only

The intent column is `VARCHAR` with a `CHECK` constraint at the DB level. Two values:
- `engagement` — relationship-building (community reply, thought-leadership post, brand-voice content)
- `commercial` — drive a buy decision (pricing page, comparison content, sales-stage outreach, paid ads)

Never invent new values. New scenarios go through validation rules on (persona_type, intent), not through new intents.

## 2. Every `emit_*` validates persona-intent before emission

`emit_content`, `emit_outreach`, `emit_technical`, `emit_operational`, `emit_paid` all funnel through `validateAndRecordEmission`, which:

1. Loads the persona by `persona_id` (404 if cross-brand).
2. Calls `validatePersonaIntent(persona_type, intent, content?, platform?)` against the matrix.
3. **If block-severity violations fire and no `override_reason` is supplied**, throws `EmissionBlockedError` and the emit halts.
4. **If `override_reason ≥ 8 chars` is supplied**, writes one audit row per blocked rule to `persona_intent_overrides` and tags the ledger row `validation_status='overridden'`.

Operational-class actions (technical, operational) pass `persona_id: null` and skip the matrix — they're persona-agnostic. The consistent contract still flows through the helper so the ledger row is uniform.

## 3. `is_branded=false` is the default scope

Every comparative read defaults to non-branded prompts. Branded comparisons measure rank ascent within a known brand context; non-branded comparisons measure recommendation likelihood when no one's heard of you. The latter is the moat metric.

Tools accept `isBranded: true` / `null` (both) but the service layer enforces tier gating: non-Grow callers requesting branded data see an `tierGated: true` flag with the competitor rows filtered out.

## 4. Share of Voice on non-branded is the North Star

The headline metric is `share_of_voice_pct` from `d_visibility_history`. Specifically, the **SoV lift attributable to a Citable-emitted action over the 30-day measurement window** is the North-Star proxy. Every aggregator, recommender, and dashboard component routes back to this.

## 5. The 4-card funnel is the outcome shape

Every ledger row's `delta_metrics` follows the same shape:
- Citation Rate
- Mention Rate
- Average Rank
- Sentiment

Plus Share of Voice + per-competitor sentiment. No tool reinvents this; everyone reads from `comparative-metrics-service.getComparativeMetrics`.

## 6. Branded competitor comparison is Grow-tier only

Defense in depth at the function level: `getBrandedVsNonbrandedBreakdown` returns `null` for branded mode when `plan_tier !== 'grow'`. The MCP tool that exposes it cannot smuggle the data past the service-layer gate.

## 7. Per-brand scan config is always respected

The brand controls `scan_stages_enabled`, `scan_models_enabled`, `scan_questions_per_stage`. Every recommendation, every aggregator weight, every outcome measurement filters by these. A brand that disabled Perplexity sees no Perplexity-derived recommendations and contributes no Perplexity signals to the cohort aggregator.

## 8. k-anonymity ≥5 on cross-brand outputs

The `cross_brand_benchmarks` table has a `CHECK (cohort_size >= 5)` constraint. The aggregator filters below-floor cohorts before insert. The table has zero `brand_id` columns — outputs are aggregates, never identifiable.

Per the v1 product call, the MCP tool that exposed cohort benchmarks is **unregistered** — cohort data is internal-only analytics. The aggregator + admin route still run.

## 9. Schema additions only, no deletions, for 1 quarter post-launch

Existing tables stay. New columns/tables add layers. Legacy code paths stay in tree, unused. The invariant prevents breaking changes for any executor coding against an older contract version.

## 10. All ledger writes try/catch-wrapped

`recordEmission()` never throws. Failed inserts log + return `null`. The user-facing action always completes — infra hiccups on the ledger side never block the platform from working.
