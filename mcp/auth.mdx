title: "Authentication"
description: "One auth method: a brand-scoped API key. How to create it, connect with it, and what it can and cannot touch."
---

Every MCP session is scoped to **one brand and one user**, resolved from your API key. The brand controls what the session can read and write; the user is recorded on every action. Neither is ever passed as a tool input.

## Create an API key

In the Citable app: **Settings → API keys → Create key**. Pick a name and an optional expiry.

The key is shown **once**, at creation — alongside a ready-to-paste client config with the key already filled in. Citable stores only a hash, so copy it then. Treat it like any other secret.

## Connect

**Local clients (Claude Desktop, Claude Code, Cursor):** use the `mcp-remote` bridge config from the [overview](/mcp/overview#connect-in-one-step) — it passes your key as a Bearer header.

**Hosted agents and direct integrations:** call the Streamable HTTP endpoint with the key in the header:

```
POST https://getcitable.com/mcp/v1
Authorization: Bearer <your_api_key>
```

Verify with the `ping` tool — it echoes the brand your session resolved to. If it's not the brand you expected, you're holding a key created under a different brand: create one under the right brand and reconnect.

## What a key can and cannot do

- A key belongs to **one brand and one user** (its creator). It cannot read or write any other brand.
- Every read is filtered to the key's brand. Every write that takes an ID resolves it and rejects with a 403 if it belongs to a different brand — for example, `approve_ticket` on another brand's ticket code fails, and the attempt is logged.
- To work with a different brand, create a key under that brand and reconnect — there is no brand-switching at runtime.

## Key lifecycle

- Revoke anytime from **Settings → API keys** — revoked keys fail immediately.
- Every successful call stamps the key's last-used time, so you can spot stale keys.
- Rotate by creating a new key, updating your config, then revoking the old one.
- Optional expiry dates auto-disable a key after the date passes.

## Rate limits

Each brand has a per-hour limit, reported in standard response headers:

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 47
X-RateLimit-Reset: 1715731200
```

The default tier allows 100 calls/hour. Higher plan tiers raise read limits; write limits apply on all tiers. Need more for a specific integration? Reach out from your account.
